## Production GraphRAG

Production GraphRAG systems face challenges beyond the research prototype: **graph freshness** (new documents arrive continuously), **cost management** (LLM extraction is expensive), **latency requirements** (users expect sub-second responses), and **observability** (why did the system return this answer?).



### Production Architecture

- **Document Ingestion** -- Queue-based async processing. New documents enter a priority queue; high-importance docs get LLM extraction, others get rule-based NER. Extraction results cached indefinitely.
- **Graph Store** -- Neptune or Neo4j with read replicas. Write through primary; queries served by replicas. Community detection runs nightly on graph delta, not full recompute.
- **Query Router** -- Classifies incoming queries as local/global/hybrid/keyword. Routes to appropriate retrieval strategy. Caches community summaries (refreshed nightly).
- **Retrieval Layer** -- Parallel execution of all relevant strategies. Results merged via RRF. Cross-encoder reranker for final precision. P95 latency target: 800ms.
- **Generation + Cache** -- LLM with retrieved context. Semantic cache: similar queries return cached responses (TTL 1hr for dynamic content). Answer traced to source graph paths.

        for node_id in affected_nodes:
            community = await self.graph_store.get_community(node_id)

      <ExpandableSection title="Cost Optimization Strategies" icon="\uD83D\uDCB0">

- **Extraction Caching** (60-80% saving) -- Cache extraction results keyed by doc_id + hash. Re-ingestion of updated docs only re-extracts changed content.
- **Tiered LLM Routing** (40-60% saving) -- Use Claude Haiku / GPT-3.5 for simple entity extraction. Claude Opus / GPT-4 only for complex relation extraction or low-confidence re-verification.
- **Community Summary Caching** (30-50% saving) -- Community summaries used in global search are expensive to generate (O(communities) LLM calls). Cache with TTL; only regenerate on dirty communities.
- **Query-level Semantic Cache** (20-40% saving) -- Semantically similar queries (cosine distance &lt; 0.05) return cached answers. Particularly effective for FAQ-style systems.
