## Enabling Vibe Coding in Enterprise SAP Environments

Vibe coding on a greenfield Next.js project is one thing. Vibe coding on a 20-year-old SAP ECC system with millions of lines of custom ABAP is a fundamentally different challenge. The principles (CLAUDE.md, plan mode, guardrails) still apply — but the context, constraints, and risks are orders of magnitude higher.

### The Legacy Vibe Coding Stack

### Step 1: Reverse-Engineer the CLAUDE.md

For legacy systems, you can't write CLAUDE.md from memory — you have to build it from the code itself:

<CodeBlock language="text" label="Building CLAUDE.md for a Legacy SAP Module">
{`# Prompt for Claude Code (after exporting ABAP code):

"Analyze the exported ABAP code in this directory and 
create a CLAUDE.md file that documents:

1. MODULE OVERVIEW
   - What business process does this code support?
   - Which SAP transactions use this code?
   - What tables are read/written?

2. ARCHITECTURE MAP
   - Key function modules and their purposes
   - Call hierarchy (who calls what)
   - Dependencies on standard SAP objects
   - Custom database tables and structures

3. BUSINESS RULES
   - Pricing logic and conditions
   - Validation rules with thresholds
   - Exception handling and error codes
   - Authorization checks

4. KNOWN RISKS
   - Deprecated function modules still in use
   - Direct database access (bypassing data layer)
   - Hardcoded values that should be configurable
   - Missing error handling

5. MIGRATION CONSTRAINTS
   - Objects incompatible with S/4HANA
   - Custom tables that need redesign
   - Interfaces that need replacement (RFC → API)"`}
</CodeBlock>

<AnalogyBox emoji="🗺️" title="Data Discovery Before Data Governance">
In MDM, you don't start with governance rules — you start with data discovery: what systems exist, what data flows where, who owns what. Legacy vibe coding follows the same pattern: discover the codebase before trying to change it. The AI-generated CLAUDE.md IS your discovery phase.
</AnalogyBox>

### Step 2: Human-in-the-Loop Validation

AI-generated understanding of legacy code is a starting point, not the final word. Critical validation steps:

<CodeBlock language="text" label="Human Validation Workflow">
{`AI generates explanation of pricing module:
"This module calculates tiered discount pricing 
based on customer segment (Z1-Z5) and order volume.
Discount tiers: 0-100 units = 0%, 101-500 = 5%, 
501-1000 = 8%, 1000+ = 12%."

Human validator (business analyst) reviews:
"Correct for segments Z1-Z3, but Z4 and Z5 have 
special override logic for government contracts 
that the AI missed. Also, the 1000+ tier was 
changed to 10% last year but the documentation 
wasn't updated — the code has 10%, the AI read 
an old comment saying 12%."

AI updates understanding:
"Corrected: Z4/Z5 government override, 1000+ tier 
is 10% not 12% (comment was stale)."

This loop continues until the human-validated 
documentation matches the actual business behavior.`}
</CodeBlock>

### Step 3: Incremental Modernization

Never modernize an entire SAP module at once. Use the same incremental approach as vibe coding:

1. **Explain first** — AI documents what the code does
2. **Validate** — humans confirm the explanation matches reality
3. **Test first** — generate ABAP unit tests for the existing behavior
4. **Transform one object** — modernize a single function module or class
5. **Verify** — run tests to confirm behavior is preserved
6. **Repeat** — move to the next object
