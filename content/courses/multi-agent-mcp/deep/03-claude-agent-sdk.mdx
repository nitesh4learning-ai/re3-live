## Claude Agent SDK: Subagents, Hooks, and Tool Search

The <JargonTip term="Claude Agent SDK">Claude Agent SDK</JargonTip> (evolved from Claude Code SDK) provides a production-ready framework for building agents with Claude. It adds three capabilities beyond raw API calls: **subagents** (agents spawning other agents), **hooks** (lifecycle callbacks), and **tool search** (dynamic tool discovery).

### Subagents: Agents Spawning Agents

<CodeBlock>
{`from anthropic import Anthropic

client = Anthropic()

def run_research_subagent(topic: str) -> str:
    """
    Spawn a focused subagent for research.
    The parent agent delegates a specific task,
    the subagent completes it independently.
    
    Like a Director delegating a data quality analysis
    to a senior analyst — clear scope, independent execution,
    results roll up to the parent.
    """
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=2048,
        system=(
            "You are a research specialist. "
            "Thoroughly research the given topic. "
            "Return structured findings with sources."
        ),
        messages=[{'{'{'}'}
            "role": "user",
            "content": f"Research this topic: {'{'{'}'}topic{'}'}"
        {'}'}]
    )
    return response.content[0].text


def run_analysis_subagent(research: str) -> str:
    """
    Second subagent analyzes the research output.
    Each subagent has its own context window and 
    system prompt — they're isolated specialists.
    """
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=2048,
        system=(
            "You are a strategic analyst. "
            "Identify patterns, risks, and opportunities."
        ),
        messages=[{'{'{'}'}
            "role": "user",
            "content": f"Analyze these findings:\\n{'{'{'}'}research{'}'}"
        {'}'}]
    )
    return response.content[0].text`}

<AnalogyBox emoji="🔧" title="Tool Use = API Orchestration">
Claude's tool use is conceptually identical to MuleSoft API orchestration — the agent discovers available tools (APIs), selects the right one for the task, formats the request, and processes the response. The difference is the agent decides which tools to use based on the conversation, rather than following a pre-defined integration flow.
</AnalogyBox>

### Best Practices for Multi-Agent Tool Design

1. **Narrow tools over broad ones**: `validate_tax_id(id)` beats `process_supplier(data)` — narrow tools are easier for the LLM to select correctly
2. **Clear descriptions**: The LLM picks tools based on their description string. Write descriptions like documentation, not code comments
3. **Schema precision**: Use `enum` for known value sets, `required` for mandatory fields, `description` for every property
4. **Error handling**: Tools should return structured error messages the agent can reason about and retry
